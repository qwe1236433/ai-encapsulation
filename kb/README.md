# 知识库（Wiki）— 渐进式说明

这里采用 **Karpathy 式「可链接的 Markdown 知识库」** 的方向，但**不一步到位**：先保证 **数据 → 实验 → 工厂** 跑通，你再按需往 `kb/` 里加页、加链接。

## 原则

1. **真值仍在数据层**：笔记样本、批次、实验数字以 **SQLite / CSV / Git忽略下的产物** 为准；`kb/` 只承载 **解释、索引、决策记录**，不写「唯一数据源」。
2. **人工确认优先**：新页、改结论、补链接，默认 **你或过一眼** 再当团队共识；暂不启用「无人值守 agent 改全库」。
3. **从小开始**：先只维护本 README + 少量索引页，避免空壳目录强迫症。

## 小红书数分：科学 → 稳 → 自动化（收敛目标）

**只做小红书**时，演进顺序固定为下面三层；**不跳层**（避免「自动化了但不可信」）。

| 顺序 | 含义 | 落地抓手（仓库内） |
|------|------|-------------------|
| **1. 科学** | 口径可写清、可复现、对外结论有 **digest / provenance / warnings** | `labels_spec`、`verify_features_labels_spec`、**`features_export_provenance.json`**、`feed_quality_metrics.warnings`、**[评估与晋升基线.md](评估与晋升基线.md)** §2.1 |
| **2. 稳** | 数据形态够支撑指标：**多轮 merge、多 digest、时间可解析、正例够门禁** | 定时 **`merge-xhs-feed`** / **`continuous-xhs-ingest -MergeOnly`**；**`FLOW_API_FEED_HEALTH_LABELS_SPEC`** 与训练同一份；`feed_ingest_health` 可选 **fail** |
| **3. 自动化** | 在 1+2 成立的前提下加编排：**digest 触发 export → train → eval → 快照**；或 **单次一键**串全部步骤 | 长驻：**`continuous-xhs-analytics.ps1`**；单次：**`scripts/xhs-research-oneclick.ps1`**（含关键词轮换、归档、`digest_comparison_report`）；eval 规则 **`merge_mediacrawler_patch_from_eval`**；晋升仍 **人工决策**（见评估与晋升基线 §4–5） |

自动化 **不** 替代：晋升基线、改契约、对外写结论——这些保持 **人过一眼**。

## 建议演进顺序（可与业务并行）

| 阶段 | 做什么 | 何时开始 |
|------|--------|----------|
| 0 | 跑通爬取 → `samples.json` → bench/一键流程；`research/` 里能导出特征、可选训练基线 | **现在** |
| 1 | 已备好三页：**[INDEX.md](INDEX.md)**（全库索引）、**[数据流与文件落点.md](数据流与文件落点.md)**（路径写死）、**[跑通检查清单.md](跑通检查清单.md)**（按 A→D 勾选）；你按需改路径、打勾即可 | **现在就能用** |
| 2 | 按主题拆页：`数据从哪来`、`实验怎么记结论`、`工厂链各步含义`，页脚挂 **「关联：research/…」** | 你有空时 |
| 3 | 再考虑 Obsidian / 自动摘要 / 向量检索入口（与 Wiki 并存） | 页数 >10或检索痛苦时 |

## 链接约定（可选）

- 链到仓库内文档：用 **相对路径**，例如 `[实验模板](../research/EXPERIMENT_REPORT.md)`。  
- 涉及数字结论：页末加一行 **`数据来源：`**（实验编号、CSV 路径或「待补」）。

## 与现有文档的关系

- **操作规程**：根目录 `使用说明书.md` 等继续保留。  
- **科学记录**：`research/EXPERIMENT_REPORT.md`、`schema_notes.md`。  
- **架构**：`项目技术架构说明.md`（全栈总览）；`小红书爆文研究-现实条件架构与路线图.md`（业务阶段）；`小红书数据与工厂-架构技术指南.md`（数据接入）。  
- **`kb/`**：逐渐承担 **「为什么这样设计、索引、团队共识」**，不替代上述文件。
