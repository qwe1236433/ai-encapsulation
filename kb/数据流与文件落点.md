# 数据流与文件落点（具体版）

> 默认仓库根目录：**`D:\ai封装`**（你若换盘符，把下面路径整体替换）。  
> 作用：回答「数据从哪来、合并后在哪、谁读、别和什么搞混」。

## 1. 爬虫原始数据（你本机）

- **常见位置**（由 `.env` 里 `FLOW_API_MEDIACRAWLER_JSONL` 或你自己约定）：例如  
  `D:\MediaCrawler\data\xhs\jsonl`  
- **形态**：多条 `.jsonl` 或 JSON，**字段名因爬虫版本而异**（title/desc/liked_count 等）。

## 2. 合并进工厂 Feed（仓库内）

- **脚本**：`scripts\export_to_xhs_feed.py`  
  - 可选 **`--dedupe none|key|content`**；Flow API / `.env` 中 **`FLOW_API_EXPORT_DEDUPE`** 与此一致。  
  - 可选 **`--validate-mode none|report|warn|fail`** 与 **`--validate-schema`**（默认 Schema 见 `scripts\schemas\xhs_feed_item_v1.schema.json`；可选 `pip install -r scripts\requirements-feed-tools.txt` 启用 **jsonschema**）；Flow 可读 **`FLOW_API_EXPORT_VALIDATE_MODE`**、**`FLOW_API_EXPORT_VALIDATE_SCHEMA`**。  
  - 可选 **`--dry-run`**：只跑合并与校验统计，**不写** `samples.json` / digest（可不传 `--out`）。  
  - 可选 **`--health-gate-mode`**（`none|report|fail`）与 **`--health-spec`**：对归一后的条目统计**健康度**（可选字段覆盖率、`published_at` 可解析比例与时间跨度、按 labels_spec 的正例数等），报告默认落 **`research/runtime/feed_ingest_health.json`**；`fail` 且未过门禁时**不写** `samples.json`。见 `scripts/feed_ingest_health.py`、`scripts/feed_ingest_health.example.json`。`merge-xhs-feed.ps1` 未设置 `FLOW_API_FEED_HEALTH_GATE_MODE` 时默认 **`report`**。  
  - 可选 **`--digest-out`**：写入 `xhs_feed_digest_v1`（输出文件 **sha256**、`merge_stats`、时间戳），便于批次审计。  
  - 可选 **`--batch-id`**：写入 digest；**Flow API** / **`merge-xhs-feed.ps1`** 可读环境变量 **`FLOW_API_FEED_DIGEST_OUT`**、**`FLOW_API_FEED_BATCH_ID`**（与 `api/main.py` 合并步骤一致）。  
- **定时或手跑包装**（读 `.env` 里的 `FLOW_API_*`）：`scripts\merge-xhs-feed.ps1`  
- **计划任务 + 日志**：`install-merge-xhs-feed-scheduled-task.ps1`（默认每 120 分钟跑 `scripts\merge-xhs-feed-with-log.ps1`，追加写入 `logs\merge-xhs-feed.log`）  
- **典型输出**（单文件模式）：  
  `openclaw\data\xhs-feed\samples.json`  
- **含义**：已是 **JSON 数组**，每条接近工厂用的 `title_hint / body_hint / like_proxy / sop_tag / emotion_tag`（归一逻辑与 `openclaw\xhs_factory._normalize_external_sample` 对齐）。

## 3. 容器里怎么读（Docker）

- Compose / `.env` 里常把样本路径指到容器内，例如：  
  `XHS_FACTORY_SAMPLES_PATH=/app/data/xhs-feed/samples.json`  
- **要点**：宿主机文件要通过 **volume挂载** 到容器内上述路径，否则容器里「看不到」你的 `D:\...`。  
- **可选**：`XHS_FACTORY_BASELINE_JSON` 指向 `train_baseline_v0.py` 产出的 JSON。  
  - **宿主机**：可放在 `openclaw\data\artifacts\baseline_v0.json`（与 `OPENCLAW_DATA_DIR` 挂载一致）。  
  - **容器内路径示例**：`/app/data/artifacts/baseline_v0.json`（在 `.env` 里把变量设为该路径）。  
  - 预测时与 `linear_clamp_v1` **blend/replace**；`/process` 的 `params` 可传 **`like_proxy_hint`**或 **`like_proxy`**（整数 ≥1）以覆盖默认 assumed like，仅影响 **baseline_lr_v0**。  
- **校验**：`python scripts\verify_features_labels_spec.py --features research\features_v0.csv --labels-spec research\labels_spec.json` 核对 CSV 中 `y_rule` 与契约阈值是否一致。

## 4. 研究用特征表（可选）

- **脚本**：`scripts\export_features_v0.py`  
- **标签阈值**：推荐 **`--labels-spec research\labels_spec.json`**（从 `research\labels_spec.example.json` 复制）；或用 `--viral-threshold`。  
- **批次列**：可选 **`--batch-id`**、环境变量 **`EXPORT_FEATURES_BATCH_ID`**、或 **`--feed-digest`**（`samples.digest.json`）写入 **`batch_id` / `feed_digest_sha256`**；基线训练仍只读 `title_len, body_len, log1p_like, y_rule`。  
- **推荐**：合并后生成 digest 的前提下，特征导出加 **`--verify-samples-digest`**，防止样本与 digest 错配；训练侧默认**禁止多批次混表**，见 `schema_notes.md`。  
- **输出**（默认）：`research\features_v0.csv`（默认被 `.gitignore` 忽略，勿当「已提交数据集」）  
- **含义**：从 `samples.json` 抽 **长度、log1p 赞、标签**；`y_rule` 需阈值（命令行或 `labels_spec`）才有。

## 5. 生成结果（文案导出）

- **目录**：`outputs\xhs-runs\*.txt`  
- **来源**：跑通 Hermes 链 + bench 导出后产生（详见 `使用说明书.md`）。

## 6. 和「数据库 / Wiki」的分工（你后面会做）

| 东西 | 放哪 | 记住 |
|------|------|------|
| 原始事实、批次、可 SQL 的东西 | 将来 **SQLite 等**（规划中） | 可复现、可 join |
| 解释、索引、决策过程 | **`kb/`** | 不替代原始表 |
| 语义检索（以后） | 向量库（以后再上） | 入口用，不是唯一真值 |

## 7. 一图读懂（文字版）

```text
MediaCrawler(jsonl 目录)
    → export_to_xhs_feed.py
    → openclaw/data/xhs-feed/samples.json
    → Docker 内 OpenClaw 读 XHS_FACTORY_SAMPLES_PATH
    → xhs_factory 挖掘 / 预测 / 生成
    → outputs/xhs-runs/*.txt

并行（研究）：
samples.json → export_features_v0.py → research/features_v0.csv → train_baseline_v0.py → research/artifacts/*.json
```

---

**你要改的时候**：只改「第 1 节」里你的真实爬虫路径 + 核对 `samples.json` 是否真存在。
